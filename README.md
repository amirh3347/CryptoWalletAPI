# CryptoWalletAPI

## درباره پروژه

این پروژه یک پیاده‌سازی ساده اما Production-Oriented از یک سیستم کیف پول (Wallet System) است که تمرکز اصلی آن روی:

- جلوگیری از Double Spending
- مدیریت Race Condition
- حفظ Atomicity تراکنش‌ها
- ثبت Ledger به صورت Immutable
- پشتیبانی از Idempotent API

هدف این پروژه نمایش نحوه طراحی صحیح یک سیستم مالی با در نظر گرفتن چالش‌های همزمانی و سازگاری دیتابیس است.

## سیستم مدیریت کیف پول:

### قابلیت ها

- ایجاد کیف پول با واحد های متفاوت ارزی
- واریز (Deposit)
- برداشت (Withdraw)
- انتقال بین کیف پول‌ها (Transfer)
- ثبت immutable تراکنش‌ها در Ledger
- مدیریت همزمانی (Concurrency Safe)
- Idempotent API
- تست‌های همزمانی و ایزولیشن دیتابیس

## تصمیمات معماری (Architectural Decisions)

در این پروژه تصمیمات زیر اتخاذ شده است:

### استفاده از Transaction Ledger Immutable
تمامی تراکنش‌ها در جدول TransactionLedger ثبت می‌شوند.
ویژگی‌ها:
- رکوردها پس از ایجاد قابل ویرایش نیستند
- حذف رکوردها در سطح دیتابیس ممنوع شده است
- Immutable بودن در سطح Database enforce شده است

هدف:
- Auditability کامل
- جلوگیری از دستکاری تاریخچه مالی

### چرا Ledger قابل ویرایش یا حذف نیست؟

در سیستم‌های مالی واقعی، تاریخچه تراکنش‌ها نباید تغییر کند.

به همین دلیل:
- update و delete در سطح Model ممنوع شده
- تغییر رکوردها Exception ایجاد می‌کند

این تصمیم باعث می‌شود:
- Audit Trail کامل داشته باشیم
- امکان دستکاری داده‌ها کاهش یابد

### مدیریت Race Condition
برای جلوگیری از Double Spending و منفی شدن موجودی:
- استفاده از select_for_update()
- اجرای عملیات داخل transaction.atomic()
- بررسی موجدی پس از تراکنش با موجودی قبلی و مبلغ تراکنش برای اطمینان از ACID بودن تراکنش

### چرا از select_for_update استفاده شد؟
در سیستم‌های مالی، مهم‌ترین خطر Race Condition است.
اگر دو درخواست همزمان برداشت ثبت کنند، بدون Lock کردن ردیف دیتابیس، ممکن است موجودی منفی شود.
به همین دلیل از:
- transaction.atomic()
- select_for_update()
استفاده شده تا اطمینان حاصل شود فقط یک درخواست در لحظه می‌تواند موجودی را تغییر دهد.
نتیجه:
- سیستم در برابر race condition ایمن است
- موجودی منفی نمی‌شود

### Idempotency در API
در دنیای واقعی ممکن است به دلیل قطع اینترنت، کلاینت یک درخواست را چند بار ارسال کند.
برای جلوگیری از انجام چندباره یک تراکنش:
- هر درخواست دارای idempotency_key است
- این کلید در Ledger یکتا است
- در صورت ارسال مجدد، همان نتیجه قبلی برگردانده می‌شود
این ویژگی برای سیستم‌های مالی حیاتی است.

### تفکیک لایه Service
منطق بیزینسی در لایه Service پیاده‌سازی شده است:
- deposit()
- withdraw()
- transfer()

View فقط وظیفه دریافت و اعتبارسنجی درخواست را دارد.

مزایا:
- تست‌پذیری بالا
- جداسازی مسئولیت‌ها
- نگهداری آسان‌تر

### استفاده از Celery برای پردازش‌های پس‌زمینه
برای بررسی تراکنش‌های با مبلغ بالا:
- ارسال Task به Celery
- استفاده از transaction.on_commit() برای جلوگیری از اجرای Task قبل از commit

### بهینه‌سازی Queryها
- استفاده از select_related
- جلوگیری از N+1
- تعریف Index روی فیلدهای کلیدی
